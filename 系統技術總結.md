## 一、開發流程（Development Process）

### 1.1　四階段瀑布式開發

| 階段 | 期程 | 主要產出 | 完成時間 |
|:----:|:----|:--------|:--------:|
| **Phase 1**　需求分析 | 決標後 1 個月 | 系統分析設計規劃書、資料庫設計書、系統設計審核報告 | 2023 / 10 |
| **Phase 2**　原型系統開發 | 決標後 3 個月 | 原型系統開發審核/測試報告、程式流程與功能說明書 | 2023 / 12 |
| **Phase 3**　模型優化與測試 | 決標後 8 個月 | 模型優化與測試審核/測試報告（含 15 項修訂） | 2024 / 05 |
| **Phase 4**　系統整合與結案 | 決標後 12 個月 | 模型開發與測試報告書、操作與訓練手冊、保固證明書（含 13 項修訂） | 2024 / 09 |

### 1.2　迭代測試流程

```
   TASA 端測試人員              NTHU 開發人員
        ┃                            ┃
        ▼                            ▼
   執行系統功能測試  ───→  回報問題  ───→  判斷是否為系統問題
                                            ┃         ┃
                                          (是)       (否 / 已修正)
                                            ▼         ▼
                                        進行修改  →  回送測試人員確認
                                            ▼
                                        確認無誤 → 關閉問題
                                        仍存在  → 回到開發端
```

### 1.3　四種測試層級

| 測試類型 | 執行者 | 目的 |
|:--------|:------|:-----|
| **單元測試**（Unit Testing） | NTHU 開發端 | 隔離驗證最小可測試單元（函式、方法） |
| **整合測試**（Integration Testing） | NTHU 開發端 | 驗證模組整合後可正確協同運作 |
| **系統測試**（System Testing） | TASA 端 | 確認整體系統符合需求、功能正確 |
| **回歸測試**（Regression Testing） | 雙方 | 確保修改不影響原有功能 |

### 1.4　關鍵調優里程碑

| 問題類別 | 數量 | 代表性修訂 |
|:--------|:----:|:----------|
| 求解效能調優 | 多項 | CPLEX 參數：8 Threads、Aggressive Gomory/Cliques Cuts、Best-Estimate Search |
| 模型可行性 | 多項 | BigM 值統一設為 ≥ 1,000,000；Duty Cycle 改以活動開始時間判斷 |
| 跨 session 排程 | 1 項 | GAT/GOHOME 生成只依 X-band 時間篩選，不限 session |
| 檔名管理 | 多項 | 檔名處理統一移至排程後，避免新舊排程混合衝突 |
| 資源利用最佳化 | 1 項 | 以後處理微調，將活動前移以充分利用 X-band contact |

## 二、系統架構（System Architecture）

### 2.1　系統定位

```
┌─────────────────────────────────────────────────────────────┐
│                  TASA 衛星操控中心架構                       │
│                                                              │
│   ┌──────────────┐         ┌─────────────┐                  │
│   │ MCC          │         │ MOC         │                  │
│   │ (Mission     │ 衛星指令 │ (Mission    │  上傳指令        │
│   │  Control     │ ───────→ │  Operations │ ──────→  衛星    │
│   │  Center)     │         │  Center)    │                  │
│   │              │         │  EPOCH 平台 │                  │
│   │  ┌────────┐  │         └─────────────┘                  │
│   │  │ MPSS   │  │                                          │
│   │  │ ★本案  │  │                                          │
│   │  └────────┘  │                                          │
│   └──────────────┘                                          │
└─────────────────────────────────────────────────────────────┘
```

本案開發的新模組整合於 **MPSS**（任務排程規劃系統）內，取代原本以 ILOG 平台撰寫之引擎，產出之衛星指令最終透過 MOC 的 **EPOCH 操控平台**上傳至衛星。

### 2.2　四層程式架構

| 層級 | 套件名稱 | 用途 |
|:----:|:--------|:-----|
| **(1) 資料存取層** | `jpa` + `dao` | 與資料庫互動的實體映射與資料存取物件 |
| **(2) 活動物件層** | `activity` | 10 種活動類別（含 Misc）的屬性與限制定義 |
| **(3) 求解模型層** | `schedule.model` | `AbstractModel`、`FS8AModel`，封裝 CPLEX 模型 |
| **(4) 參數設定層** | `config.properties` | 排程權重、優先序、Duty Cycle 等可調參數 |

### 2.3　活動物件設計

所有活動類別共同擁有四個方法：

| 方法 | 用途 |
|:-----|:-----|
| `constraints()` | 該活動**尚未排程時**所需的限制條件 |
| `scheduledConstraints()` | 該活動**已排程時**所需的限制條件 |
| `update()` | 求解後更新於資料庫的資訊 |
| `unschReason()` | 當活動無法排入時指出無法排入的原因 |

**拍照系列活動**（RSI Record / Playback / Delete）另含記憶體與檔名管理方法：

- `setSubActivity()`：依拍照類別指定子類別
- `cal_recorderdelta()` / `getSSRConsume()`：計算 SSR 記憶體佔用
- `initFileName()` / `getFilenames()` / `getFilenameConsume()`：檔名管理
- 同 `arlentryid` 的 REC → PBK → DEL 之間透過上述方法進行資料傳遞

### 2.4　求解引擎

| 項目 | 規格 |
|:-----|:-----|
| 求解器 | **IBM ILOG CPLEX** |
| 模型類型 | MILP（混合整數線性規劃） |
| 主要模型 | `FS8AModel.defineModel()` — 排程最佳化 |
| 次要模型 | `FS8AModel.unscheduledModel()` — 無法排入原因搜索 |
| 求解參數 | Threads=8、Gomory Cuts=Aggressive、Cliques Cuts=Aggressive、NodeSelect=Best-Estimate |

### 2.5　可擴充性

文件第 9 節「新增活動與其對應的限制式」提供新建活動的標準作業流程，未來若 TASA 有新業務需求（如新增 ROTATE 類活動），可依下列步驟擴充：

1. 在 `Abstract_Model` 列舉新增 Activity Type
2. 在 `addIloNumVar` 宣告該活動執行與否的變數
3. 在 `FS8AModel` 將變數加入目標函式
4. 在 `activity` 套件新建活動類別並實作 4 個基本方法
5. 在 `IlogActivityBuilder` 完成建立

## 三、資料交換介面（Data Exchange Interface）

### 3.1　資料介面總覽

```
                ┌─────────────────┐
                │  MPSS 既有 DB    │
                │ （延用 FS5 設計） │
                └────────┬────────┘
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼
        【讀入】       【模型】      【寫回】
       匯入資料表    CPLEX 求解   更新資料表
            │            │            │
            └────────────┴────────────┘
                         │
                         ▼
                  Logs (INFO/ERROR)
```

### 3.2　輸入端：讀取資料表

| 資料表 | 內容 | 用途 |
|:-------|:-----|:-----|
| `ACTIVITY` | 所有衛星活動清單 | 已排程→限制；未排程→排程對象 |
| `CONTACT` | 地面站接觸時窗 | 建立 Monitoring、GAT/GOHOME 活動的依據 |
| `SESSIONSITEXREF` | Session 與 Site 對應關係 | 取得本 session 可使用的站台 |
| `UNAVAILTIMERANGE` | 站台不可用時段 | 排除無效時窗 |
| `SITE` | 地面站基本資訊（含 prefer 權重） | 站台選擇與權重設定 |
| `SESSIONS` | 軌道圈次資訊 | 區分 cold/hot 排程範圍 |
| `RSIIMAGINGREQUESTS` | RSI 影像拍照請求 | 拍照系列活動排程依據 |
| `ECLIPSE` | 衛星日食時段 | SPL Procedure 限制依據 |
| `SATELLITEPS` / `RECORDERPS` | 衛星本體與記錄器參數 | 記憶體容量計算 |

### 3.3　輸出端：寫回資料表

| 資料表 | 更新內容 |
|:-------|:--------|
| `ACTIVITY` | 已排入活動的 `scheduled` 欄位 → 1；新增系統創建之 Monitoring、GAT、GOHOME 活動 |
| `EXTENT` | 新增有被排入的活動執行區段資訊 |
| `SESSIONS` | 該 session 的 `coldscheduled` 欄位 → 1（cold scheduling 才更新） |

### 3.4　使用者介面參數（API 層）

各功能由使用者端透過參數呼叫，主要參數類型：

| 參數 | 說明 |
|:-----|:-----|
| `session id` | 指定要排程的 session 範圍 |
| `activity id(s)` | 指定要考慮的活動清單 |
| `contact id` | 指定地面站接觸時段（-1 = 不指定） |
| `station name` | 指定地面站名稱（空字串 = 不指定） |
| `site id(s)` | 指定可用地面站清單 |
| `start time` / `end time` | 限定排程時間區間 |
| `move date` / `adjust date` | 微調活動目標時間 |

### 3.5　系統 Log 介面

格式：`[類型] 日期 時間 內容`

| 欄位 | 內容 |
|:-----|:-----|
| 類型 | `[INFO]` 運行資訊 / `[ERROR]` 錯誤訊息 |
| 日期 | YYYY-MM-DD |
| 時間 | HH:MM:SS |
| 內容 | 訊息本文 |

供 TASA 端維運人員追蹤系統運行狀態與除錯。

### 3.6　設定檔介面

`config.properties` 可由使用者編輯，主要參數類別：

- 各活動類型優先順序
- 時間窗口寬度（如 GAT 抓取範圍 181 秒）
- Duty Cycle 上限（RSI Record 8 min、Playback 10 min）
- 各地面站 prefer 權重值
- 目標函式各項權重值
- CPLEX 求解器參數

## 四、主要功能（Main Functions）

### 4.1　兩大排程模式

#### Cold Scheduling（冷排程）

> 用於**長期、整體**規劃，以 S-band contact 為基礎建立 Monitoring 活動。

**執行流程**：

1. 建立 CPLEX 求解模型實例
2. 讀取 ACTIVITY Table（已排程→限制；未排程→對象）
3. 讀取 CONTACT / SESSIONSITEXREF / UNAVAILTIMERANGE / SITE Table → 建立 Monitoring 活動
4. 為各活動加入「可單獨考量」之限制式
5. 加入「無法單獨考量」之限制式
6. 定義目標式 → 呼叫 CPLEX 求解
7. 更新 ACTIVITY、EXTENT、SESSIONS Table

#### Hot Scheduling（熱排程）

> 用於**動態調整**，以 X-band contact 為基礎建立 GAT 與 GOHOME Maneuver。

**執行流程**：與 Cold Scheduling 類似，但建立的是 GAT / GOHOME 活動，後處理會刪除無用的 GAT/GOHOME（即兩者之間無任何 RSI Playback 者）。更新 ACTIVITY、EXTENT Table。

### 4.2　五項微調功能（Hot Scheduling 延伸）

| 功能 | 用途 | 主要參數 |
|:-----|:-----|:--------|
| **runUIReHot** | 經使用者調整後嘗試將未排入活動排入 | `session id`、`station name`、`activity id(s)`、`contact id` |
| **addResourceChange** | 移動已排入的 RSI Delete 位置 | `session id`、`activity id`、`move date` |
| **adjustActivity** | 調整已排程活動的執行時間 | `session id`、`contact id`、`activity id(s)`、`adjust date` |
| **runUIReFilterHot1** | 在指定時間區間內嘗試排入未排程活動 | `start time`、`end time`、`site id(s)`、`activity id(s)` |
| **runUIReFilterHot2** | 不限時間區間，於 session 內排入未排程活動 | `session id`、`site id(s)`、`activity id(s)` |

> **共同特性**：所有微調功能皆「不影響已排程活動」為前提，若無法滿足使用者需求則不進行調整。

### 4.3　活動類型與限制（10 種活動）

| 類型 | 活動 | 核心限制 |
|:----:|:-----|:--------|
| **姿軌控** | Attitude Maneuver | 與其他 Maneuver / REC / PBK 不可重疊 |
|  | Orbit Maneuver | 與所有其他活動不可重疊 |
|  | GOHOME Maneuver | 每 50 分鐘充電一次；PBK 後須執行 |
|  | GAT Maneuver | PBK 前須執行 |
| **監控** | Monitoring | 需在 S-band 時窗執行 |
| **拍照系列** | RSI Record | 每軌 ≤ 8 min；需記憶體足夠 |
|  | RSI Playback | 11.5° X-band 時窗；每軌 ≤ 10 min；需先有同 ID 的 REC |
|  | RSI Delete | PBK contact fade + 6 sec；需 REC + PBK 完成 |
| **SPL** | SPL Procedure | 10° S-band + Eclipse 中執行 |
|  | SPL Playback | S-band 接觸；需先有同 ID 的 SPL Procedure |

### 4.4　雙模型設計

| 模型 | 目的 | 目標式 |
|:-----|:-----|:-------|
| **主排程模型** | 求出最佳排程結果 | **最大化**：可排入活動總數 + 優先序權重 + 站點權重 − PBK/DEL 完成時間 |
| **原因搜索模型** | 當無解時找出衝突來源 | **最小化**：違反限制條件的總數 |

當主模型無解時，系統會切換到原因搜索模型，自動偵測並回報違反的限制式類別（如：Attitude 與 GAT 重疊、Duty Cycle 超限等），供使用者調整資源或活動後再次嘗試排程。

### 4.5　可擴充功能

依需求書，系統已實作以下需求項目：

| 需求項目 | 實作方式 |
|:--------|:--------|
| 可對衛星軌道、衛星及地面站資源進行規劃 | CPLEX 目標式（Goals）求解 |
| 可解決排程衝突 | CPLEX 限制式（Constraint）設定 |
| 可對資源進行管理 | 參數皆由 DB 獲取，可由 Client 調整 |
| 可同時支援多衛星操作 | 可設定多衛星同時操作上限 |
| 可支援長期與短期規劃 | 參數變化產出不同情境結果 |
| 可對福衛八號活動進行排程 | 為 FS-8 客製化限制與目標 |
| 可支援科學資料收集 | 排程結果回饋至 DB |
| 可支援發射規劃 | 因應發射內容調整排程 |
| 可支援緊急規劃 | 處理活動必需/無法執行、SSR 變化等 |
| 可進行快速重新規劃 | 依上一版本排出差異最少結果 |
